# Prototype MVP - Rewritten Submission (Reality-Aligned)

## 1) MVP Description (2-4 sentences)
We built an interactive Streamlit prototype for a Managed Onboard Compute Payload (MOCP) safety workflow. The MVP is designed for CubeSat teams and space-software builders who need to evaluate onboard fault-handling behavior before investing in a full mission stack. It simulates power and temperature telemetry, supports fault injection, and shows safety-state transitions across NOMINAL, THROTTLE, SAFE, and RECOVER. The assumption under test is that transparent transition logic (clear reasons for every state change) increases operator trust and improves incident triage.

## 4) Iteration, Learning, and Next Steps
### What we changed in this MVP iteration
- Added explicit state machine logic and transition logging with reasons.
- Added optional hysteresis/minimum dwell controls to reduce state flapping.
- Added timeline/event views and CSV export for post-run analysis.
- Added service blueprint workflow inside the app so non-technical reviewers can follow the pilot process.

### What worked better than expected
- The state model is easy to follow in a visual interface.
- Event timeline + “Why am I in this state?” makes fault behavior interpretable.
- The app is lightweight and runs locally with minimal setup.

### What still feels risky or unclear
- Simulator realism is limited; this is not hardware-in-the-loop and not flight validation.
- Adoption depends on whether real operators trust the abstractions and thresholds.
- We still need direct usability evidence from target users.

### Next steps
- Run 5-10 real user sessions and capture structured feedback.
- Add a scenario preset library (e.g., eclipse-heavy, thermal-stress, sensor-failure) for repeatable tests.
- Add side-by-side comparison views for baseline vs hysteresis behavior.
- Use testing evidence to refine product scope and pilot-readiness criteria.
